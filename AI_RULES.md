# AI 에이전트 코딩 지침서 (AI_RULES.md)

이 문서는 AI 에이전트가 본 프로젝트의 코드를 작성하고 수정할 때 발생하는 실수를 예방하고 품질을 극대화하기 위한 행동 지침입니다. 에이전트는 작업을 시작하기 전 이 지침을 반드시 확인해야 합니다.

속도보다는 신중함과 안정성을 우선시합니다. 단, 사소하고 명확한 작업의 경우 합리적인 판단 하에 유연하게 적용하십시오.

---

## 1. 코딩 전에 먼저 생각하기
추측하여 코드를 작성하지 마세요. 모호하거나 확실하지 않은 부분은 임의로 결정하지 말고 명확히 밝혀야 합니다.

작업을 실행하기 전에:
- **가정 검증**: 세운 가정을 개발 시작 전에 사용자에게 명확히 설명하세요. 확실하지 않다면 먼저 질문하세요.
- **대안 제시**: 여러 해석이나 구현 방식이 가능하다면, 임의로 선택하지 말고 각각의 장단점과 대안을 함께 제시하세요.
- **단순화 제안**: 더 효율적이고 간단한 방법이 있다면 제안하고, 필요하다면 기존 요구사항에 대해 피드백을 전달하세요.
- **이해 중단**: 이해하기 어려운 부분이 생기면 작업을 멈추고 헷갈리는 부분을 구체적으로 정리하여 사용자에게 도움을 요청하세요.

---

## 2. 단순성(Simplicity) 최우선
문제를 해결하는 데 필요한 최소한의 코드만 작성하고 불필요한 추측성 코드는 절대 포함하지 마세요.

- **최소 기능 주의**: 요청받은 범위 이외의 추가 기능이나 확장은 절대 구현하지 마세요.
- **과도한 추상화 금지**: 일회용 코드나 단순 로직에 불필요한 디자인 패턴이나 추상화 레이어를 도입하지 마세요.
- **유연성 배제**: 요청되지 않은 미래 대비용 "유연성"이나 "환경설정 옵션"을 미리 설계하지 마세요.
- **간결한 구현**: 200줄의 코드를 50줄로 줄여서 동일한 품질로 구현할 수 있다면, 반드시 더 간결한 방식으로 재작성하세요.
- **선임 리뷰 자문**: 스스로 "경험 많은 시니어 엔지니어가 이 코드를 보았을 때 지나치게 복잡하다고 지적할까?"라고 질문해 보고, 그렇다면 즉시 구조를 단순화하세요.

---

## 3. 수술적 변경 (Surgical Changes)
변경이 꼭 필요한 코드만 수정하세요. 본인의 변경 사항에 대한 뒷정리도 철저히 해야 합니다.

기존 코드를 편집할 때:
- **무관한 변경 금지**: 수정하려는 대상 외에 주변 코드, 주석, 포맷팅을 임의로 "개선"하지 마세요.
- **리팩토링 경계**: 정상적으로 작동하고 있는 기존 비즈니스 로직을 동의 없이 임의로 리팩토링하지 마세요.
- **스타일 일치**: 본인의 코드 작성 선호도와 다르더라도 프로젝트의 기존 코드 스타일과 변수 명명 규칙을 철저히 따르세요.
- **미사용 코드 발견 시**: 작업 영역 주위에 기존에 방치된 미사용 코드를 발견하더라도 함부로 삭제하지 말고 사용자에게 먼저 보고하세요.

작업 중 고아 파일이나 코드가 발생하는 경우:
- **정리 작업**: 본인의 작업 결과물로 인해 더 이상 사용하지 않게 된 `import`, 변수, 함수 등은 깨끗하게 제거하세요.
- **기존 미사용 코드 보존**: 요청받은 작업과 무관하게 기존에 이미 존재하던 미사용 코드는 임의로 삭제하지 마세요.
- **목표 일치**: 수정되는 모든 라인은 오직 사용자의 요청 사항 구현과 직접적으로 연관되어야 합니다.

---

## 4. 목표 중심적 실행 (Goal-Oriented Execution)
구체적인 성공 기준을 정의하고, 해당 기준이 완벽히 충족될 때까지 검증을 거듭하십시오.

작업 단위를 검증 가능한 구체적 목표로 변환하세요:
- **"예외 처리 추가"** → "잘못된 입력 케이스에 대한 테스트 코드를 먼저 작성하고, 이를 모두 통과하도록 예외 로직 작성"
- **"버그 수정"** → "해당 버그를 재현하는 최소 조건의 테스트를 작성하고, 수정 후 이 테스트가 정상 통과하는지 확인"
- **"리팩토링"** → "변경 전과 후의 실행 결과 및 테스트가 완벽히 동일하게 동작하는지 비교 검증"

여러 단계를 거쳐야 하는 복잡한 작업의 경우, 구현 전에 항상 검증 절차를 포함한 명확한 단계별 계획을 공유하세요.
1. [작업 단계] → 검증 방법: [체크 포인트]
2. [작업 단계] → 검증 방법: [체크 포인트]
3. [작업 단계] → 검증 방법: [체크 포인트]

불필요한 작업과 구조적 복잡성을 배제하고, 구현 후 오류 수습이 아닌 구현 전 논리적 검토와 질문을 통해 품질을 보증합니다.
